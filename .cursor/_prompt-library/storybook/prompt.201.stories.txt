핵심요구사항)
1) 기본 구조 (CSF)
- `title`, `component`, `tags`, `args`, `argTypes`, `parameters.layout`를 포함할 것.
- 기본 확인용 `Playground` 스토리를 포함할 것.

2) 컨트롤 구성
- 컴포넌트의 핵심 props를 컨트롤에 연결할 것. (피그마·API에 실제로 있는 축만 넣는다.)
  - 예: `variant`, `type`, `state`, `size`, `disabled`, `fullWidth`, `nativeType`
  - (아이콘 슬롯이 있으면) `showLeftIcon`, `showRightIcon`
- `ReactNode` 타입 props(예: `leftIcon`, `rightIcon`)는 샘플 기본값을 두고, `argTypes`에서 `control: false` 또는 `table.disable`로 컨트롤 테이블에서 숨긴다.
- 아이콘 슬롯을 불린으로 토글할 때는 `render`에서 조건에 따라 실제 노드가 전달되도록 구현한다. 아이콘 소스는 **팀 아이콘 라이브러리 우선**이며, 정적 SVG·`<img>`는 스펙에 있을 때만 쓴다.

3) 필수 스토리 세트
- `Playground`
- `Variants` (variant 비교 — 해당 시)
- `Types` (fill/outline 등 타입 비교 — 해당 시)
- `States` (상태 비교 — 컴포넌트에 state·시각 축이 있을 때)
- `Sizes` (size 비교 — 해당 시)
- 필요 시 `FullWidth`, `Filled`, `Disabled` 등 실사용·경계 케이스 추가
- 피그마에 없는 조합은 만들지 않는다. (UI 구현 시 정한 축 설계와 모순 없게.)

4) `state`·프리뷰와 실제 상호작용(해당 시)
- 본체 CSS가 가상 클래스(`:hover` 등)로 동작하면, 스토리의 `state` prop은 **고정 시각 프리뷰용**일 수 있다. 스토리 `description`·주석으로 의도를 명시한다.
- 루트에 `data-state` / `data-variant` 폴백이 있으면, 스토리에서 그 값이 CSS와 맞는지 확인한다.
- `disabled === true` 등 **props 우선순위**가 있으면 스토리 설명에 적는다.

5) 검증 가능성
- 스토리만으로 주요 UI·동작을 팀원이 빠르게 점검할 수 있어야 한다.
- 프로젝트에서 정의한 variant 의미(색·용도)를 스토리에서 구분 가능해야 한다.
- 정적 이미지/SVG·`<img src="...">` 사용 시 해당 레포의 ESLint·접근성 규칙을 따른다(필요 시 규칙에 따른 예외 주석 등).
