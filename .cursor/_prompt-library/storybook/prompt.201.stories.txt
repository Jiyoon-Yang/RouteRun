가이드) 스토리 검증 범위가 넓거나 팀 공용 기준을 남겨야 할 때는 이 Long 템플릿을 사용.

아래 조건을 모두 적용하여 컴포넌트의 Storybook 스토리를 구현할 것.
구현 결과를 체크리스트로 반환할 것.

==============================================

조건-커서룰)
- 아래 커서룰을 적용하고, 작업 종료 시 적용 결과를 체크리스트로 반환할 것.
  - @01-common.mdc

==============================================

조건-파일경로)
- 참고 TSX: [예: src/commons/components/{component-name}/index.tsx]
- 참고 CSS: [예: src/commons/components/{component-name}/styles.module.css]
- 구현 STORIES: [예: src/commons/components/{component-name}/index.stories.tsx]

==============================================

핵심요구사항)
1) 기본 구조 (CSF)
- `title`, `component`, `tags`, `args`, `argTypes`, `parameters.layout`를 포함할 것.
- 기본 확인용 `Playground` 스토리를 포함할 것.

2) 컨트롤 구성
- 컴포넌트의 핵심 props를 컨트롤에 연결할 것.
  - `variant`, `type`, `state`, `size`
  - `disabled`, `fullWidth`, `nativeType`
  - (아이콘형 컴포넌트면) `showLeftIcon`, `showRightIcon`
- `ReactNode` 타입 props(예: `leftIcon`, `rightIcon`)는
  - 샘플 기본값 제공
  - `argTypes`에서 `control: false`로 비활성화

3) 필수 스토리 세트
- `Playground`
- `Variants` (variant 비교)
- `Types` (fill/outline 비교)
- `States` (default/hover/active/disabled 비교)
- `Sizes` (size 비교)
- 필요 시 `FullWidth` 등 실사용 케이스 스토리 추가

4) 검증 가능성
- 스토리만으로 컴포넌트의 주요 UI/동작을 팀원이 빠르게 점검할 수 있어야 함.
- 현재 프로젝트에서 정의한 variant 의미(예: primary/secondary/tertiary 색상 의미)를 스토리에서 명확히 확인 가능해야 함.

체크리스트(출력 포맷)
- [ ] 지정 경로에 스토리가 구현됨
- [ ] 필수 스토리 세트가 모두 포함됨
- [ ] 핵심 props가 컨트롤에서 조작 가능함
- [ ] (해당 시) 아이콘 불린 토글이 컨트롤에서 동작함
- [ ] 팀원이 스토리만 보고 컴포넌트 상태를 검증할 수 있음
