핵심요구사항) 컴포넌트는 실제 입력 가능한 Input으로 구현할 것.
            - 프리뷰/스토리북 전용 강제 상태 구조는 사용하지 말 것
            - JS props로 상태를 고정하지 말 것

핵심요구사항) props는 HTML 표준 호환을 우선할 것.
            - React.InputHTMLAttributes<HTMLInputElement>를 확장할 것
            - placeholder, value/defaultValue, onChange, disabled, readOnly, required, name, id, autoComplete를 그대로 지원할 것
            - 커스텀 props는 최소화하고 아이콘 슬롯·시맨틱 상태(success, error)만 허용할 것 (error는 DOM의 aria-invalid로 반영)

핵심요구사항) 상태는 가상클래스(Pseudo-class)와 표준 속성으로 구현할 것.
            - default: 기본 상태
            - hover: :hover
            - focus: :focus 또는 :focus-within
            - disabled: :disabled, [aria-disabled='true']
            - error/invalid: [aria-invalid='true'] (필요 시 :invalid 보조 사용)
            - success: .success 클래스 또는 [data-status='success'] 속성으로 처리할 것
            - filled: :not(:placeholder-shown) 패턴으로 처리할 것

핵심요구사항) placeholder/값 동작은 HTML 기본 규칙을 유지할 것.
            - placeholder를 value처럼 강제로 렌더링하지 말 것
            - 값이 없으면 placeholder, 값이 있으면 입력값을 표시할 것
            - focus 상태에서 placeholder 표시 규칙을 임의로 변경하지 말 것

핵심요구사항) 상태 우선순위를 CSS에서 명확히 보장할 것.
            - 우선순위: disabled > error/invalid > success > focus > hover > filled > default
            - 우선순위가 높은 상태가 낮은 상태를 덮어쓰도록 선택자를 구성할 것

핵심요구사항) hover 적용 제한을 CSS 선택자로 명시할 것.
            - 예시: input:hover:not(:disabled):not([aria-invalid='true']):not(.success):not([data-status='success']) { }
            - disabled, error/invalid, success 상태에서는 hover 스타일이 적용되지 않게 할 것

핵심요구사항) 접근성(Accessibility)과 표준 속성을 함께 만족할 것.
            - 에러 표현 시 error(boolean) 또는 표준 aria-invalid를 설정 가능하게 할 것
            - disabled 상태는 input의 disabled 속성으로 제어하고 필요 시 aria-disabled를 병행할 것
            - placeholder는 보조 정보로만 사용하고 라벨 대체 수단으로 사용하지 말 것

핵심요구사항) 스타일 범위는 Figma primary 단일 필드로 제한할 것.
            - variant/size/state prop은 두지 말 것
            - 기본 클래스(styles.base) + 가상클래스/속성 선택자 중심으로 구성할 것
            - 디버그 전용 data-state는 기본 구현에서 제외할 것

핵심요구사항) 아이콘 슬롯은 팀 공통 규칙을 따를 것.
            - FieldLucideIcon 기반으로 렌더링할 것
